chickenchomper.com

The site you are viewing is a basic web server implementation using Python, HTML, CSS, and JavaScript. It was designed to be simple and quick to develop, so Flask was chosen as the web framework. Self-described as a 'microframework', Flask certainly felt lightweight with a single-command install and a 7-line 'Hello World' web server. Making the site openly available was as easy specifying port 80 and running as root.

Hosted on this site are personal projects, simple web games, and various resources. I have put the source code for my personal projects on GitHub, and plan to incorporate demos of some, if not all, of them into this site. The web games are coded in Flash as well as HTML5 and use a sql database to record statistics.

I am actually in the process of porting the source code from PHP to Python, so this site doesn't have everything it should yet. You can view the new site's source here, although pushes will likely lag a little behind what is running

Tetris puzzle solver

Inspired by a puzzle competition, this program takes in a rectangle matrix and fits the correct tetris pieces to cover all the squares. However, instead of working with a blank slate, there are a few restrictions:

  • every cell must be a part of a tetris piece
  • some of the piece boundaries are given
  • no two copies of the same piece may share an edge (reflections count as different pieces, rotations do not)

Url aliaser

This is a 'work in progress' project between me and one of my friends. In essense, it is an exercise in implementing a basic TinyURL-like service from scratch to identify and tackle some of the associated challenges.

For example, one of the problems discussed was regarding how to generate the alias url. In the best case, the url would be as short as possible, responses would be fast (reduce collisions), and the algorithm should work when multiple servers are generating alias urls. The solution we are planning on implementing will involve partitioning the 'alias url space' and assigning a range of urls to use to each front end server. This way, each front end server can generate aliases sequentially locally. They will checkpoint globally where they are in that range periodically to help deal with fault tolerance. A locking mechanism will be used to ensure no two servers are given the same ranges.

The lightweight Spark Java web framework was chosen for speed of development. Although we want to think of performance at the cluster level, we don't actually care about the local server's throughput. The backend consists of DynamoDB table with an index for 2-way url lookups. DynamoDB local is used for integration tests so we do not have to actually use a real table.
